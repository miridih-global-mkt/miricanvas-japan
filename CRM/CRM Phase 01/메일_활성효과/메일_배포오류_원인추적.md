# JA1(1차실험) 모든 metric z=0 원인 추적

분석일 2026-05-19. 대상: `세그먼트별_메일성과_compact.html`에서 JA1(`Global온보딩JA260407`) 모든 metric이 0/0~ 표시되는 현상.

---

## 결론 (TL;DR)

**JA1 raw 데이터는 정상이고 신호도 존재한다. L2_v4 ranking 파이프라인이 JA1에 한해 모든 cell의 z를 0으로 잘못 출력한 버그.**

- 기존 분석문 메모("D168h_ge3 zero division 의심") → **틀린 진단**. D168h_ge3는 JA1 raw에서도 trt 0.025, ctrl 0.023으로 NON-zero
- JA1 × COMPANY × aip × **LPU168h**: 손계산 Welch z = **−0.73** (파이프라인 출력 0)
- JA1 × COMPANY × aip × **L168h_ge3**: 손계산 Welch z = **−1.63** (의미 있는 음효과, 파이프라인 출력 0)
- 즉 raw 데이터는 “aip 메일이 COMPANY 유저의 일주일 내 3일 이상 로그인 비율을 낮추는 약신호”를 가지고 있는데, 산출물엔 0으로 출력됨

원인은 (a) raw 데이터 부재 (b) zero division (c) canvas 매핑 누락 (d) trt/ctrl 정의 오류 — **어느 것도 아님**. 정확한 메커니즘은 파이프라인 코드가 폴더에 없어 단정 불가하나, 가장 유력한 가설은 **"weekly cohort SE 계산 시 weeks 부족 또는 SE=0 케이스에 대한 잘못된 fallback (z를 0으로 강제)"**.

## Step 4 — 수동 Welch z 산출 (이 cell)

```
Welch z = (trt_mean − ctrl_mean) / sqrt(sd_trt²/n_trt + sd_ctrl²/n_ctrl)
```

**LPU168h** (Control sd=1.031, aip sd=0.927):
- diff = -0.049
- SE = √(0.927²/406 + 1.031²/434) = √(0.002117 + 0.002449) = **0.0676**
- **z = −0.726**

**L168h_ge3** (Control sd=0.2148, aip sd=0.1626):
- diff = -0.0213
- SE = √(0.1626²/406 + 0.2148²/434) = √(0.0000651 + 0.0001063) = **0.01310**
- **z = −1.63**

**DPU168h** (Control sd=0.732, aip sd=1.064):
- diff = +0.054
- SE = √(1.064²/406 + 0.732²/434) = √(0.002787 + 0.001234) = **0.0634**
- **z = +0.85**

(Welch는 단순한 two-sample z이지만 ranking 파이프라인이 사용했을 weekly cohort SE보다 보통 SE가 더 좁아 z가 더 큼. 즉 두 방법 모두 위 cell의 z를 0이 아니라 절대값 0.7~1.6 수준으로 산출해야 함.)

→ L2_v4_results.csv의 z=0은 **파이프라인 산출 오류**.

## 가설별 판정

| 가설 | 판정 | 근거 |
|---|---|---|
| H1: D168h_ge3 zero division | ❌ 부정 | trt/ctrl 모두 ~2.3% (10명/430명) NON-zero |
| H2: 관측 기간 짧아 신호 없음 | ❌ 부정 | LPU168h Welch z=-0.73, L168h_ge3 Welch z=-1.63 — 의미 있는 신호 존재 |
| H3: Canvas naming 매핑 누락 | ❌ 부정 | L2_v4_results.csv에 JA1 1,000행 모두 존재 (그러나 z=0으로 채워짐) |
| H4: Treatment/Control 정의 오류 | ❌ 부정 | is_control='Y'와 'N' 명확 구분, COMPANY 모수 정상 |
| **H5(신규): 파이프라인의 SE/z fallback 버그** | ✅ **유력** | 산출물 패턴(JA1 *전체* z=0) + raw 신호 존재 → 파이프라인이 "weeks<X" 또는 "SE=0 fallback"으로 JA1 cell을 통째 zero-out 처리한 것으로 추정 |

**H5 메커니즘 가능성**:
- 파이프라인이 weeks ≥ 4 필터 적용 → JA1(3주)은 z=0으로 강제
- 또는 weekly SE 산출 시 한 주라도 SE=0이면 fallback으로 모든 z=0 (rare metric인 D168h_ge3의 어느 한 주 분산=0 가능성)
- 또는 JA1 처리 코드 분기에 명시적 skip 조건

확정은 파이프라인 코드를 다시 만들거나 원작자 확인 필요. 그러나 **사실 자체는 명확**: JA1 raw 데이터는 정상이고 신호도 있다.

## Fix 방향 (사용자 컨펌 필요)

### Option A — 단순 two-sample z 재계산 (권장)

JA1 4 step × 7 user_type × 12 (또는 40개) metric에 대해 `temp.ejjeong_temp.crm_global_new`에서 SQL로 직접 Welch z 산출 → 별도 `L2_v4_JA1_redo.csv` 생성 → compact CSV 7개에서 JA1 row 갱신 → HTML 재빌드.

- 장점: 즉시 가능. 우리가 SQL 통제 가능
- 단점: 다른 캔버스가 사용한 weekly cohort SE와 방법이 달라서 z 스케일이 약간 다를 수 있음 (대체로 단순 z가 SE 좁아 z 크게 나옴)

### Option B — Weekly cohort SE 재구현

다른 캔버스와 동일 방법(weekly cohort empirical SE)으로 JA1만 재계산. 원작자가 사용한 정확한 공식 필요.

- 장점: 다른 캔버스 z와 비교 가능
- 단점: 정확한 공식 추정 필요, weeks=3로 weekly SE 안정성 ↓

### Option C — JA1 분석 제외 + 명시적 caveat

HTML에서 JA1 row를 "관측 기간 짧고 산출 신뢰도 미정"으로 회색 처리하거나 별도 섹션. raw 데이터엔 신호가 있지만 다른 캔버스와 동등 비교 어려움.
