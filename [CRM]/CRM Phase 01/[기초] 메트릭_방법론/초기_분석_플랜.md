# 캔버스 별 성과: "가장 좋은 메트릭" 선정

## Context

`temp.ejjeong_temp.crm_global_new` 테이블 기반 CRM Global Canvas Dashboard의 **(개괄) 캔버스 별 성과** 탭은 캔버스×스텝별로 20여 개의 메트릭을 한 번에 보여준다.
하지만 메트릭이 너무 많고 일부는 단위(×100 스케일)·분산·해석이 섞여 있어 **"무엇을 보고 결정할지"가 명확하지 않다**.

사용자(CRM 담당)의 목표는 명확해졌다:
- **무엇을 결정하는가**: Control 대비 Treatment의 uplift/downlift가 **우연이 아닌 지속적 경향**으로 잡히는지
- **궁극 목적**: 1·2·4주 뒤에도 **계속 쓰는 유저(=로그인 또는 디자인 생성)** 만들기. **디자인 생성이 primary, 로그인은 편집·다운로드 등을 포괄하는 간접 proxy**
- **대상**: 신규 유저 온보딩/활성화
- **결정 주기**: 미정 (이 plan에서 추천)

테이블에는 결제·구독·D+30 retention 컬럼이 없다(이미 확인됨). 따라서 메트릭 선정은 **테이블 내 행동 신호를 어떻게 조합·압축할지의 문제**이며, "북극성 직접 측정"이 아닌 **"북극성의 가장 좋은 proxy"** 를 고르는 문제다.

---

## 1. "좋은 메트릭"의 5가지 기준 (이 맥락 기준)

| # | 기준 | 왜 중요한가 |
|---|---|---|
| **S1** | **북극성 정렬 (Validity)** | 1·2·4주 retention과 인과·상관 높아야 함. 단기 행동이 장기 사용으로 이어지는 신호여야 함. |
| **S2** | **통계적 안정성 (Signal-to-Noise)** | Control vs Treatment 차이가 표본 분산보다 커야 일별/주별 lift가 "흔들리지 않고" 잡힘. **분산이 작은 metric일수록 적은 표본으로 유의차 감지 가능.** |
| **S3** | **민감도 (Sensitivity)** | 캠페인 효과가 작아도 움직임을 포착. floor/ceiling 효과 없어야 함 (예: open_rate은 이미 높으면 더 안 올라감). |
| **S4** | **시간 적합성 (Timeliness)** | 결정 주기 안에 충분한 표본이 쌓여야 함. 168h metric을 24시간마다 보는 건 신선도 vs 신뢰도 trade-off. |
| **S5** | **해석·게이밍 저항 (Interpretability)** | 팀·이해관계자가 단위·방향을 직관적으로 이해. 이상치 한두 명에 흔들리지 않음(outlier robustness). |

**Rate형(0/1 평균) vs Per-user형(연속값 평균)의 본질적 차이**
- Rate: 분산 = p(1−p)/n → 표본이 많으면 매우 안정. 해석 쉬움. **하지만 floor/ceiling 가능**.
- Per-user: heavy-user(소수가 디자인 100개 만드는 패턴) 때문에 **분산이 폭증**. 평균이 흔들림. **상한/winsorize로 보정 필수**.

**핵심 발견**:
- **DRate168h**: S1~S5 모두 최상위. **Primary 후보 1순위**.
- **상한DPU**: depth(얼마나 깊이 쓰는지)를 outlier-robust하게 측정. DPU 계열 중 유일하게 통계 안정.

- **×100 스케일 메트릭(LPU, DPU 모두)**: 단위 함정. 표기 fix 또는 raw 평균으로 재정의 필요.

## 3. 놓친/추가 권장 메트릭 (현재 대시보드에 없음)

테이블 컬럼은 이미 모두 노출되어 있지만, **조합·재정의로 더 좋은 메트릭**을 만들 수 있다.

### 3-1. ⭐ **D7≥3** — "7일 내 디자인 3개 이상" (신규 메트릭)
- 정의: `CASE WHEN design_count_168h >= 3 THEN 1 ELSE 0 END`의 평균
- 왜 좋은가:
  - 기존 `D>3`(누적)의 코호트 편향 제거 — **모든 유저에게 동일한 7일 관측창**
  - "한 번 써보고 떠나는" 유저 제외 → **retention 의도가 진짜로 있는 활성화 정의**
  - 사내 활성화 표준이 되기 좋음
- **이 plan의 최우선 추천**

### 3-2. **L7 & D7** — "7일 내 로그인 AND 디자인 모두" (활성화 결합)
- 정의: `login_168h_yn = 1 AND design_168h_yn = 1`
- 디자인만 본다 ≠ 로그인만 본다. 둘 다 한 유저가 진짜 활성 유저 (사용자 정의: "쓰는 유저 = 로그인 + 디자인")
- DRate168h에 비해 floor가 낮아 lift 감지 민감도 ↑

### 3-3. **Activation-Curve AUC (D1·D3·D7 누적 디자인 도달의 면적)**
- 정의: `(DRate24h + DRate72h + DRate168h) / 3` 또는 가중평균
- 단일 시점의 도달이 아닌 **속도+최종도달 모두** 반영. 시각화 시 retention curve의 면적.
- 단일 숫자로 캔버스 비교에 강함.

### 3-4. **Capped LPU / DPU (raw 평균, 10 cap)** — 단위 통일
- 현재 LPU·DPU에 곱해진 `* 100`을 제거하고 raw 평균으로 재정의. 상한DPU와 통일.
- 단위 함정 제거. 해석 가능성 ↑.

### 3-5. **per-user_type stratified uplift** (이미 데이터 있음, 시각화만)
- user_type(BUSINESS/COMPANY/EDUCATION)별로 Control vs Treatment lift를 **누적 일별** 추적
- "user_type 별로 또는 무관하게 봐서 매일매일 경향성이 잡히는지" 라는 사용자의 요청에 직결

### 3-6. **Sample Ratio / SRM guardrail**
- Control vs Treatment 유저 수 비율이 의도(예: 50:50)에서 벗어나면 lift 자체가 편향. 메트릭이 아니라 **검증 가드레일**로 항상 같이 표시.

## 4. 최종 추천 — 2-layer 메트릭 체계

### Layer 1: **Primary (단일 KPI)**
> ⭐ **DRate168h** (7일 내 디자인 ≥1 도달률)
- 이유: S1~S5 종합 최상. binary라 통계 안정. 신규 활성화 표준 정의. 이미 정의되어 있어 즉시 사용.
- **사용자의 "디자인 생성이 primary"** 원칙에 가장 직접 부합.

### Layer 2: **Triangulation (1차에서 lift 잡힌 후 검증)**
1. **D7≥3** (신규 정의) — *활성화 깊이*: 1회 사용 vs 반복 사용
2. **상한DPU** — *활성화 강도*: 사용량 평균 (outlier 보정)
3. **LRate168h** — *간접 proxy*: 디자인 안 만들어도 편집·다운로드한 유저까지 포함

### Guardrail (방향이 반대로 가면 alert)
- **user (count distinct)**: SRM 체크
- **open_rate**: 메일 deliverability 이상 감지
- **Long-term: LRateTotal / DRateTotal**: 7일 lift가 장기 reversal 없는지 1~2주 후 후행 검증

### 결정 주기 추천: **2-tier cadence**
- **일별(매일)**: DRate168h의 **7일 rolling cohort lift** 추적. 즉, "오늘 기준, 가입 8~14일 전 유저들의 Control vs Treatment DRate168h 차이". 매일 새 코호트가 들어와 trend가 보임.
- **주별(매주)**: 누적 Control vs Treatment 비교 + statistical test (z-test for proportions, 표본 N≥수천이면 신뢰 가능). 캔버스 on/off 결정.
- **월별**: LRateTotal·DRateTotal로 7일 결과의 장기 유효성 검증.

> **"24h를 매일 보는 것"은 비추**: 표본이 작아 noise 큼. 168h를 매일 rolling으로 보는 게 신호 안정성·신선도 균형.

## 5. 검증 방법 (How to verify)

1. **재정의 메트릭 SQL 작성**: D7≥3, L7&D7, Capped LPU/DPU를 SQL CTE로 추가
   ```sql
   -- 예: D7≥3
   AVG(CASE WHEN design_count_168h >= 3 THEN 1.0 ELSE 0.0 END)
   ```

2. **분산 비교 (S2 검증)**: 동일 캔버스에 대해 각 메트릭의 표본 표준편차 / 평균 = CV(변동계수) 계산. CV가 낮은 메트릭일수록 적은 표본으로 lift 감지 가능.

3. **상관 분석 (S1 검증)**: 캔버스×유저 단위로 `DRate168h ↔ LRateTotal`, `DRate168h ↔ design_count_total` 상관계수. 0.5+면 단기 활성화가 장기 사용을 잘 예측.

4. **일별 cohort uplift simulation**: 지난 4주 데이터로 매일 DRate168h Control vs Treatment lift를 plot. **lift 방향 일관성(같은 부호 비율)** 측정. 70%+면 "noise 아닌 경향" 충족.

5. **A/A test (sanity)**: Control을 random하게 둘로 쪼개 lift가 0 근처에서 분산되는지 확인. 메트릭 false-positive 비율 점검.
