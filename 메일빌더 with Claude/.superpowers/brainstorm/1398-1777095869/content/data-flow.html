메일 1통이 JSON으로 어떻게 생겼고, 사용자 액션이 어떻게 전달되는지.

① JSON 스키마 (메일 1통)

{
  "id": "20260425-143012-a1b2c3",
  "title": "D7 폰트 변경 안내",
  "number": 4,
  "createdAt": "2026-04-25T14:30:12+09:00",
  "lastSavedAt": "2026-04-25T15:42:00+09:00",
  "language": "ja",
  "sections": [
    {
      "id": "sec-001",
      "type": "hero",                  // 라이브러리의 6종 중 하나
      "padding": null,                // null = 카탈로그 디폴트 사용
      "modules": [
        { "id": "mod-001", "type": "img", "src": "https://...", "alt": "logo", "width": 120 },
        { "id": "mod-002", "type": "h1", "text": "MiriCanvasメルマガ" }
      ]
    },
    {
      "id": "sec-002",
      "type": "body",
      "padding": { "top": 40, "bottom": 20 },  // 위/아래만 override (좌우는 디폴트)
      "modules": [
        { "id": "mod-003", "type": "p", "html": "안녕하세요, <strong>ejjeong</strong>님" },
        { "id": "mod-004", "type": "cta", "label": "지금 사용해보기", "href": "https://miricanvas.com/..." }
      ]
    }
  ]
}
핵심: id, type, padding(override), modules[] — 단순. 모든 섹션·모듈에 id 부여 (드래그앤드롭·Undo 추적용). padding이 null이거나 누락되면 카탈로그 원본 그대로.

② 사용자 액션의 흐름

한 번의 텍스트 수정이 일어나는 과정
1
사용자가 좌측 편집화면 모듈 내 텍스트 입력 (<p> 모듈 안의 글자 수정)
↓
2
editor.js가 입력 캡처 → 200ms debounce → 메모리 내 JSON 트리 업데이트 (currentMail.sections[1].modules[0].html = "...")
↓
3
Undo 스택에 push (이전 트리 스냅샷, 최대 30개) + preview.js가 우측 화면 다시 렌더 (비주얼/코드 동시)
↓
4
상단바 저장 버튼이 활성화 + 표시 ("저장 안 됨"). 닫기 버튼은 비활성화 유지
↓ 사용자가 저장 버튼 클릭
5
PUT /api/mails/{id} → 서버: ① 기존 파일을 mails/.backup/로 이동(타임스탬프 부여) ② 새 JSON 파일 저장 ③ 백업은 최근 3개만 유지
↓
6
저장 완료 → 닫기 버튼 활성화. Undo 스택은 그대로 유지 (저장 후에도 Ctrl+Z 가능)

③ 미저장 변경사항 처리

🚪 브라우저 닫기 / 새로고침
브라우저 기본 컨펌 팝업: "이 페이지를 떠나시겠습니까? 저장되지 않은 변경사항이 있습니다."
🗂 다른 탭 클릭 (소개/비교)
제작탭 상태는 메모리에 보존. 탭 전환 시 경고 없음. 다시 제작탭 돌아오면 그대로
🚫 다른 메일의 편집 버튼
스펙대로 차단 팝업: "현재 편집 중인 메일을 닫은 다음, 편집할 메일을 선택해주세요"
↶ Ctrl+Z (Undo)
최대 30 단계. 메일 닫으면 스택 초기화. 저장 여부와 무관하게 동작
