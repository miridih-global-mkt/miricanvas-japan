# 성과분석 파이프라인 v7 — 원칙·계산 방법

작성: 2026-05-19. v6를 outlier 보정(Winsorize 99%) 추가해 개선한 버전.

## 변경 요약 (v6 → v7)

v6는 user-level Welch z를 사용했으나 **count·ratio metric의 outlier에 취약**했다. 일부 메일의 효과가 헤비 유저 한두 명에 의해 왜곡되는 현상 (특히 디자인 개수, 로그인 일당 디자인 수). v7는 9개 outlier-prone metric에 **Winsorize 99%** 를 적용해 보완.

## Winsorize 적용 대상 (9개 metric)

| 카테고리 | metric |
|---|---|
| 디자인 개수 | DPU24h, DPU72h, DPU168h, DPU_total |
| 로그인 일당 디자인 수 (ratio) | Stickiness_24h, Stickiness_72h, Stickiness_168h, Stickiness_total |
| 누적 로그인 일수 | LPU_total |

Winsorize 안 함 (34개):
- Binary 비율류 (LRate*, L*_ge2/ge3, DRate*, D*_ge2/ge3, Activ_AUC, LD*, LDeep*, LRateReturn, DRateReturn): 값 0~1로 bounded → outlier 자체 없음
- 일/주 자연 상한 (LPU24h: 0~1, LPU72h: 0~3, LPU168h: 0~7): 자연 cap 있음

## Winsorize 정의

각 (canvas × user_type × step) cell 안에서 PERCENTILE_APPROX(metric, 0.99)을 산출.  
그 cell의 상위 1% 사용자 값을 99 percentile 값으로 cap → AVG·STDDEV 계산.

```sql
WITH p99 AS (
  SELECT canvas_key, user_type, step_key, PERCENTILE_APPROX(metric, 0.99) p_metric
  FROM crm_global_new
  GROUP BY canvas_key, user_type, step_key
)
SELECT AVG(LEAST(metric, p99.p_metric)) ...
```

## 점검 근거 (outlier 영향 검증)

산출 전 점검에서:
- LPU24h/72h/168h: 영향 미미 (max |Δz|<1.0, sign-flip 3~4%)
- DPU24h/72h/168h/total: **영향 큼** (max |Δz|=3~5, sign-flip 10~22%)
- Stickiness 4종: **심각** (max |Δz|=110+, sign-flip 16~33%)
- LPU_total: 영향 작지만 sign-flip 12% — 보수적 차원 포함

## Z·consistency 계산 (v6와 동일)

- **z = (trt_mean − ctrl_mean) / √(sd²_trt/n_trt + sd²_ctrl/n_ctrl)** — Welch two-sample
- **consistency** = signup_time 주별 cohort 부호 일치율 (양쪽 주별 cell ≥10 충족 주만)
