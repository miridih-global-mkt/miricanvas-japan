Braze vs Framer — B2B 랜딩페이지 도구 선택

01 랜딩페이지 도구 선택 기준
좋은 랜딩페이지 도구를 고르는 기준은 세 가지였다.
① 구현이 쉬운가?
HTML 직접 편집, AI 생성 지원, MCP 연동 등 수작업 없이 빠르게 만들 수 있는 환경인가. 디자이너·개발자 없이 마케터 혼자 돌아가야 한다.
② 자동화·워크플로가 되는가?
폼 제출 이후 다음 단계가 자동으로 이어지는가. 자동회신 이메일 발송, 후속 캔버스 트리거 등 "제출 → 그 다음"이 연결돼야 한다.
③ 트래킹 문제가 없는가?
GA4, Airbridge 등 기존 추적 체계와 충돌 없이 연결되는가. 새로운 버그를 만들거나 기존 데이터를 오염시키지 않아야 한다.
Webflow 배제 이유
마케팅팀이 독자적으로 Webflow를 쓰면 SEO 트래픽이 분산되어 기존 미리캔버스 사이트에 악영향을 줄 수 있다. 검토 대상에서 아예 제외.
02 Braze vs Framer — 기준별 비교
기준 Braze 랜딩페이지 Framer
① 구현 용이성 드래그앤드롭 에디터, 폼 블록 네이티브
MCP 연동 없음		 노코드 기반이나 고품질 UI는 손이 감
Design Bridge MCP 연동 가능
② 자동화 ✅ Canvas 자동회신 네이티브
폼 제출 즉시 트리거 내장		 ❌ 별도 연동 필요
Braze와 연결하려면 개발 개입

팀 프로젝트에 이미 세팅됨
중복 프로필 외부 유입 시 새 프로필 생성 가능 해당 없음
결론
Braze가 현실적으로 우월한 선택이다. 자동회신이 네이티브로 연결되고, B2B 소량 문의 특성상 트래킹 정밀도보다 운영 편의가 더 중요하다. Framer는 광고 전환 최적화가 핵심인 대량 캠페인에 더 맞는다.
03 Braze 자동회신 설정
전체 흐름
초기 메일
CTA 링크를 {% landing_page_url 'LP이름' %} Liquid 태그로 생성. 이 링크를 통해 들어온 유저는 기존 프로필에 자동 연결됨.
↓
폼 제출
유저가 랜딩페이지에서 폼을 제출. Braze가 이 이벤트를 자동으로 감지.
↓
Canvas 트리거
Action-based Canvas의 Submitted a Landing Page form이 발동. 해당 LP를 지정해두면 이 LP에서만 반응함.
↓
자동회신 발송
첫 스텝의 Message(Email)이 즉시 또는 딜레이 후 발송.
설정 단계
  1. Messaging → Landing Pages에서 랜딩페이지 생성. 폼에 Email Capture 블록 포함 필수. Publish까지 완료.
  2. Canvas 생성 → Entry: Action-Based → Submitted a Landing Page form → 해당 LP 지정.
  3. 첫 스텝에 Message(Email) 추가. 구독 상태: subscribed or opted-in.
유저 식별 — 어디까지 보장되나
  • Braze 메시지 링크 경유: 기존 프로필에 자동 연결. 데이터 유실 없음.
  • 외부 유입(광고 직링크): 새 braze_id 프로필 생성. 기존 유저와 동일 이메일이어도 자동 merge 안 됨. 자동회신 자체는 입력한 이메일로 정상 발송됨.
  • 중복 해소 필요 시: /users/merge API로 두 프로필을 하나로 합침. identifier는 email / phone / external_id 중 선택.

04 GA/GTM — 실제로 필요한가
자동회신만 목적이라면
GA/GTM 전혀 불필요. Braze가 자체적으로 폼 제출을 감지해서 처리한다.
GA4 세션 연속성이 목적이라면
  • Braze 랜딩페이지는 miricanvas.com과 도메인이 다르다. GTM 코드만 넣는다고 같은 세션으로 이어지지 않음.
  • GA4 속성에서 크로스도메인 트래킹 설정을 추가해야 하는데, 이는 분석팀/DA팀 협조 필요.
  • 이 허들 때문에 현실적으로 포기하는 편이 합리적.
Airbridge 일원화 이후에도 GA는 유효한가
  • 유효하다. 두 시스템은 경쟁 관계가 아니라 병행 운영.
  • 에어브릿지 트래킹링크를 쓰면 GA와 Airbridge 모두 정상 수집됨.
  • 내부 결론: "에어브릿지 추적이 잘 돼도 GA 로우데이터를 데이터브릭스에서 활용하기 때문에 UTM값이 붙는 것이 중요." → 둘 다 필요.

05 Airbridge — 트래킹링크 vs Web SDK
역할 분리
  • 트래킹링크: 클릭 순간에만 관여. 누가 어느 광고를 클릭했는지를 기록하고 UTM 파라미터를 붙여서 목적지로 보냄. 랜딩페이지 도달 이후는 관여하지 않음.
  • Web SDK (airbridge.init()): 랜딩페이지 안에 설치됐을 때 작동. 페이지 도달을 기록하고, 이 방문을 원래 클릭과 연결함. 이벤트 코드를 추가하면 폼 제출도 기록 가능.
측정 항목트래킹링크만Web SDK 추가 시
광고 클릭✅ 수집✅ 수집
페이지 도달(뷰)❌✅ 수집
폼 제출❌⚠️ 이벤트 코드 추가 시
매체 전환 신호❌⚠️ Postback 설정 시
클릭 vs 페이지뷰 — 실질적 차이는?
  • 단일 캠페인 → 단일 랜딩 구조에서는 클릭 수 ≈ 페이지뷰 수, 차이 크지 않음.
  • Web SDK를 추가하면 여러 경로로 들어온 유입을 페이지뷰 하나로 통합해서 볼 수 있다는 편의 정도의 이점.
  • 클릭한다고 전부 페이지뷰가 찍히는 건 아니라는 점(로딩 중 이탈 등)에서 미세한 차이는 있음.

버튼 이벤트를 혼자 추가하면 왜 안 되나
  • 회사 내 이벤트 taxonomy(이벤트명 규격)가 있고, 여기에 없는 이벤트를 임의로 보내면 Airbridge·GA4 리포트에서 분류가 안 되거나 기존 집계와 섞임.
  • 표준 이벤트명(airbridge.user.signup 등)을 쓰면 괜찮지만, 이것도 B2B 문의폼에 딱 맞는 이름인지 담당자 확인이 필요.
B2B 문의폼에서 전환 최적화가 어려운 이유
  • 전환 볼륨이 작아서 매체 알고리즘이 학습할 데이터가 충분하지 않음.
  • 클릭 최적화나 랜딩페이지 조회 최적화로 운영하는 것이 현실적.
최소 실용 세팅
에어브릿지 트래킹링크 + Braze Canvas 자동회신. 커스텀코드 없이도 클릭 볼륨 파악과 자동회신을 모두 커버할 수 있다. 나머지는 복잡도 대비 이득이 작다.
07 커스텀코드 배치 — head/body 환경 기준
각 코드가 왜 거기 들어가는지부터 이해하는 것이 먼저다.
GTM 스니펫 — <head> 최상단
GTM 창고를 여는 열쇠. 이 코드 하나만 페이지에 있으면, GTM 대시보드에서 관리하는 모든 태그(GA4, Airbridge SDK 등)가 자동으로 이 페이지에서도 발동한다. 페이지가 뜨자마자 실행돼야 해서 head에 넣는다.
consent 초기화 — <head> GTM 직후
데이터 수집 동의의 기본값을 설정하는 코드. GTM이 뜨는 순간 동의 상태를 이미 알아야 하기 때문에 순서가 중요하다. 미리캔버스 기준으로는 기본값이 수집 허용(granted)이다.
Airbridge init — <head>
"이 페이지에서 Airbridge를 켜줘"라는 선언. 실행되는 순간부터 페이지뷰가 Airbridge에 기록되기 시작한다. 방문 기록 유실을 줄이려면 최대한 빨리 실행돼야 한다.
GTM noscript — <body> 첫 줄
JavaScript를 꺼놓은 환경을 위한 대비책. 현실적으로 JS를 끄는 사람은 거의 없어서 사실상 형식적인 코드다. HTML 요소이기 때문에 body 안에 넣는다.
<head>
  <!-- GTM 스니펫 -->
  <script>(function(w,d,s,l,i){ ... })(window,document,'script','dataLayer','GTM-5WHCT5P');</script>

  <!-- consent 초기화 (GTM 직후) -->
  <script>
    window.dataLayer = window.dataLayer ?? [];
    window.dataLayer.push({ event: 'consent_event' });
    window.dataLayer.push({ event: 'config_user_initialize', plan_type: 'FREE', user_id: '0' });
  </script>

  <!-- Airbridge init -->
  <script>
    /* Airbridge SDK 로더 전체 (Airbridge 문서에서 복사) */
    airbridge.init({ app: 'miricanvas', webToken: '4453b89ac0764a3d8342bfae35d70db9' });
  </script>
</head>

<body>
  <!-- GTM noscript -->
  <noscript><iframe src="https://www.googletagmanager.com/ns.html?id=GTM-5WHCT5P"
    height="0" width="0" style="display:none;visibility:hidden"></iframe></noscript>
</body>
⚠️ 코드 안전성
GTM-5WHCT5P는 miricanvas.com 공용 컨테이너다. 이 코드를 Braze 랜딩에 붙이면 컨테이너 안의 모든 기존 태그가 이 페이지에서도 발동된다. 위키에서 웹빌더에 이 컨테이너를 쓰라고 명시하므로 실제 큰 문제가 생길 가능성은 낮지만, 확신을 갖고 쓰려면 DA/분석팀에 "이 컨테이너 Braze 랜딩에 써도 되나요?" 확인 한 마디를 받는 것이 안전하다. consent 코드는 2025년 4월 기준이라 최신 버전 별도 확인 필요.

08 Braze Custom Code 블록 활용
사용 방법
  1. Edit Landing Page → 왼쪽 Build 패널 → Custom code 블록 드래그
  2. 캔버스에서 블록 클릭 → 오른쪽 속성 패널에 코드 입력
  3. CSS는 <style>...</style>, JS는 <script>...</script> 형태로 같이 넣음
화면 구성의 자유도
HTML + CSS + JavaScript로 거의 제한 없이 만들 수 있다. 단, 예쁘게 만드는 것과 Braze에 데이터를 저장하는 것은 별개의 문제다.
Braze 저장 방식 — 두 가지 경로
  • 폼 블록(Email Capture 등): 제출 시 Braze가 자동으로 프로필에 저장. 기본 검증과 제출 플로우가 포함돼 있어서 이메일·전화번호 등 표준 데이터에 적합.
  • Custom Code + lpBridge API: 커스텀 UI를 만들고 버튼 클릭 시 lpBridge로 수동 저장. 배열·중첩 구조처럼 폼 블록이 지원 못 하는 복잡한 데이터에 사용. 
// 커스텀 속성 저장
window.lpBridge.setCustomUserAttribute("survey_answer", "option_A");

// 즉시 Braze로 전송
window.lpBridge.requestImmediateDataFlush();
주의할 점
  • Custom Code의 <input>·버튼은 Braze와 자동으로 연결되지 않는다. 클릭 이벤트 처리와 lpBridge 호출을 직접 구현해야 함.
  • 이메일·전화번호 저장이나 구독 그룹 처리를 lpBridge로 완전 대체할 수 있는지는 문서에서 명확히 안내되지 않는다. 이 영역은 폼 블록을 쓰는 것이 안전.
  • 외부 유입(광고 직링크)으로 들어온 경우, 제출 시 새 프로필이 생성될 수 있다는 점은 폼 블록·Custom Code 모두 동일하게 해당.
10 한 페이지에 여러 목적의 폼을 넣을 수 있는가
기본 전제 — 한 페이지 = 한 번의 폼 제출
폼 블록(Email Capture, Input Field, Checkbox 등)을 여러 개 배치하는 건 가능하다. 그러나 이 블록들은 하나의 폼 제출로 묶인다. 문서상 "여러 개의 독립된 폼을 각각 별도 제출로 분리"하는 기능은 명확히 안내돼 있지 않다.
패턴 A — 폼 블록 + 버튼 로그로 CTA 구분
  • Button 블록의 Submit form when button is clicked를 켜면 제출 버튼이 된다.
  • 버튼을 여러 개 두어도 모두 같은 폼을 제출하는 형태 — "버튼마다 다른 필수 입력 항목"은 맞지 않는다.
  • 버튼별 구분이 필요하면: Log custom attributes or events로 cta_type = "download" 같은 값을 함께 저장하고, Canvas에서 조건 분기.
패턴 B — Custom Code + lpBridge로 완전 분리
  • 각 CTA(문의/다운로드/미팅/설문 등)를 Custom Code로 독립 UI로 구현.
  • 버튼 클릭 이벤트마다 서로 다른 키로 setCustomUserAttribute(...) + requestImmediateDataFlush() 호출.
  • 사실상 버튼마다 별도 저장처럼 구현 가능. 단, HTML/JS를 직접 짜야 하는 개발 비용이 생긴다.
운영 관점 권장
LP를 목적별로 2~3개로 분리하는 것이 가장 깔끔하다. 한 페이지에 전부 넣어야 한다면 — 폼 블록 방식으로 하나의 제출로 묶고 버튼 로그로 목적을 구분하거나, Custom Code + lpBridge로 각 섹션을 독립 위젯처럼 구현한다.
